Análisis del caso

Riesgos de calidad ⚠️

Rendimiento deficiente 🤷‍♂️

Tiempos de carga lentos, retrasos en tiempo real o cuelgues al mover tareas. Sus causas pueden ser: Consultas ineficientes, Uso excesivo de CPU, Latencia de Red

Errores en la sincronización ⚙️

Cambios no visibles en tiempo real o pérdida de datos por ediciones simultáneas. Sus causas son: Falta de manejo de concurrencia (sin CRDT o bloqueos), Fallos en el servidor (colas saturadas), Dependencia excesiva del cliente.

Problemas de escalabilidad 🧑‍💻

Colapso al pasar de cientos a miles de usuarios. Ejemplo: Sistema falla con 1,000 equipos. Sus Causas son: Arquitectura monolítica, Sin caching, Infraestructura rígida (sin autoescalado como AWS Lambda)

Falta de seguridad 🦠⚠️

Brechas en datos sensibles (planes, documentos). Ejemplo: Ataques roban o sabotean proyectos. Sus causas son: Autenticación débil (sin MFA), Datos sin cifrar (sin HTTPS o encriptación), Control de acceso insuficiente (roles mal definidos)

Deuda Técnica 🧑‍💻⚠️

Deuda técnica es el término que se utiliza para describir el resultado de tomar decisiones basadas en la velocidad por encima de todo. Estas decisiones rápidas, en tiempo real pueden hacer o deshacer las actualizaciones de software.

⚙️ Deuda técnica en Fallas de seguridad 🧑‍💻

El sistema de gestión interna permite que las sesiones de los empleados permanezcan activas indefinidamente sin cerrar por inactividad

  • ➡️ Si un usuario deja su equipo desatendido, un atacante podría acceder sin restricciones
  • ➡️ Riesgo de uso indebido de credenciales

⚙️ Deuda técnica por Error de diseño 🎨

No se implementara la autenticación multifactor (MFA) desde el inicio. Los desarrolladores deciden posponerlo porque quieren lanzar el producto rápido y se decide que se agregara en una fase posterior

  • ➡️ Riesgo de ataques de phishing y fuerza bruta
  • ➡️ Accesos no autorizados

⚙️ Deuda técnica en calidad y pruebas ⚠️

Se decide no escribir pruebas automatizadas y confiar únicamente en pruebas manuales realizadas por QA

  • ➡️ Cada nueva versión debe pasar por un proceso manual de pruebas, lo que ralentiza el ciclo de desarrollo
  • ➡️ A medida que la aplicación crece, el esfuerzo manual se vuelve insostenible
  • ➡️ QA debe probar repetidamente las mismas funcionalidades en cada actualización
  • ✅ Falta de Monitoreo y Alertas de Sincronización
  • ✅ Dificultad para identificar y resolver problemas
  • ✅ Mayor carga en el equipo de desarrollo

⚙️ Deuda Técnica Falta de Modo Offline Robusto 🥸

Los cambios hechos sin conexión pueden perderse o no aplicarse correctamente cuando el usuario vuelve a estar en línea. No hay un almacenamiento local para guardar cambios cuando no hay conexión

  • ➡️ Frustración del usuario porque los cambios desaparecen
  • ➡️ Pérdida de productividad si las tareas no se sincronizan correctamente
  • ➡️ Los usuarios migraran a otras plataformas

⚙️ Uso Ineficiente del Almacenamiento en la Nube ☁️

La app guarda demasiados datos en la nube sin optimización, lo que incrementa costos y ralentiza la sincronización

  • ➡️ Costos elevados por almacenamiento innecesario
  • ➡️ Sincronización más lenta, afectando la experiencia del usuario

⚙️ Deuda Técnica cifrado solo en transito 🔒

La app cifra la información solo cuando se transmite (por ejemplo, usando HTTPS), pero los datos almacenados en la base de datos o en backups quedan sin cifrar. Se asume que el acceso a la base de datos está suficientemente protegido

  • ➡️ Si un atacante obtiene acceso a la base de datos, podrá leer todos los datos sin restricción
  • ➡️ En caso de un fallo de seguridad

⚙️ Deuda Técnica arquitectura 🧑‍💻

La aplicación sigue un diseño monolítico, y a medida que crece, se vuelve difícil escalar componentes individuales sin afectar todo el sistema

  • ➡️ Baja escalabilidad horizontal
  • ➡️ Actualizaciones difíciles
  • ➡️ Rendimiento degradado

Aseguramiento de calidad 💎

  • Mejorar el rendimiento 🛜

    Optimizar consultas a bases de datos, Reducir uso de CPU/Memoria, Minimizar latencia red

  • Garantizar sincronización en tiempo real ⚙️

    Manejar concurrencia, Robustecer servidor

  • Asegurar escalabilidad 🥇

    Modernizar la arquitectura, Implementar caché estratégico, Autoescalado dinámico

  • Fortalecer seguridad 🔒

    Autenticación robusta, Proteger datos, Control de acceso

Medidas de control ✅

  1. Gestión y Planificación de Requisitos 🤔

    Definir claramente las funcionalidades como la interfaz, la colaboración en tiempo real, la integración con otras herramientas, el soporte multiplataforma y los criterios de calidad

  2. Metodología de desarrollo ágil ✍️

    Utilizar metodologías agiles como Scrum para iterar rápidamente, obtener feedback y ajustar los requisitos

  3. Control de calidad y pruebas 🌐

    Verificar la consistencia y actualización en tiempo real de los datos para evitar retrasos y errores en la información que se comparte

  4. Monitoreo y Logging 🕵️

    Implementar un sistema robusto de logging para analizar errores, incidentes y comportamientos inesperados de la aplicación

Medidas de Control Vitales 💎

  1. Seguridad integral 🔒

    Realizar pruebas de vulnerabilidad para detectar y corregir posibles vulnerabilidades

  2. Arquitectura escalable y modular 🛜

    Adoptar arquitecturas basadas en microservicios o componentes desacoplados, facilitando la escalabilidad

  3. Documentación y capacitación 📄

    Asegurar que todos los integrantes del equipo estén familiarizados con las herramientas, metodologías y estándares de calidad adoptados
